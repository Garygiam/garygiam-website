## Objective

Treat `GI-002` (`GaryGiam.com`) as the benchmark `GEH Reference Implementation`.

PB12 does not exist simply to improve one website.

Its purpose is to strengthen the reference implementation that future `GEH Instances` may inherit from, while preserving the constitutional rules established by `PB11`.

## Governing Baseline

The authoritative baseline for PB12 is the external PB11 package at:

- `/Users/garygiam/Desktop/PB11-evolution-framework-package`

This package governs PB12.

The `GaryGiam.com` repository is the implementation under evaluation against that baseline.

PB12 must not reconstruct PB11 from memory or from historical repository documents when the external package already defines the constitutional reference.

## Constitutional Rules To Preserve

PB12 must follow these rules throughout:

- `GEH Core` remains unchanged unless a separate GEH Core programme is authorized.
- `GaryGiam.com` may produce evidence, findings, recommendations, and reusable patterns, but it may not redefine `GEH Core` by itself.
- `Factory Candidates` may be identified, but they are not promoted during PB12.
- `GEH Core Candidates` may be identified, but they are not promoted during PB12.
- `Local Improvements` remain local unless later validated through the formal GEH promotion lifecycle.
- promotion decisions must be evidence-based, not convenience-based.

## PB12 Working Sequence

PB12 should operate in the following order:

1. Review and validate the external `PB11` package.
2. Validate PB11 recommendations and implications against the current `GaryGiam.com` repository.
3. Build one prioritized backlog grouped into:
   - `Reference Quality`
   - `Factory Candidates`
   - `GaryGiam.com Improvements`
   - `Deferred`
4. Insert `Step 3.5 â€” Promotion Review` before implementation.
5. Execute only the highest-priority `Reference Quality` improvements first.
6. Capture reusable learnings in the `Knowledge Harvest`.
7. Pause at the CEO checkpoint before moving to the next priority group.

## Step 1 â€” PB11 Review And Validation

PB12 must read every document in the approved PB11 package and extract the rules that matter for a reference implementation.

This review should validate:

- the PB11 promotion lifecycle
- evidence thresholds
- asset graduation logic
- factory qualification rules
- release discipline
- the distinction between `GEH OS`, `GEH Core`, `GEH Factories`, and `GEH Instances`

The result of this step is not implementation.

It is an evidence-backed validation frame for evaluating the repository.

## Step 2 â€” Repository Validation

PB12 must validate the current `GaryGiam.com` repository against the PB11 baseline and record findings from actual repository evidence.

If any PB11 expectation conflicts with current repository reality:

- record it as a finding
- classify the maturity of the finding appropriately
- do not assume the external PB11 expectation is wrong
- do not assume the repository is already compliant

The repository should be treated as a real governed instance whose evidence may inform future GEH evolution, but not as a self-authorizing source of constitutional truth.

## Step 3 â€” Prioritized Backlog

PB12 must build a single backlog with four grouped sections:

### Reference Quality

Use this for items that make `GaryGiam.com` stronger as a benchmark reference implementation for future GEH inheritance.

These items improve clarity, repeatability, evidence quality, governance readability, asset structure, and reusable benchmark quality.

### Factory Candidates

Use this for items that show signs of becoming reusable GEH systems with bounded inputs, outputs, and operating rules, but do not yet qualify for promotion.

These items are identified, not promoted.

### GaryGiam.com Improvements

Use this for improvements that strengthen the local website or repository, but do not yet justify reference or factory treatment.

These remain local by default.

### Deferred

Use this for items blocked by missing evidence, missing authorization, cross-context validation needs, or clear out-of-scope status.

## Step 3.5 â€” Promotion Review

Before any implementation begins, every backlog item must be explicitly classified as one of:

- `Reference Quality`
- `Factory Candidate`
- `GEH Core Candidate`
- `Local Improvement`
- `Deferred`

This stage exists to prevent category drift and to ensure PB12 strengthens the wider GEH ecosystem without confusing local usefulness for canonical readiness.

### Promotion Review Rules

- every item must have one primary classification
- classification must be based on repository evidence plus PB11 constitutional rules
- items may be strong locally without being reusable enough for GEH-wide promotion
- a `GEH Core Candidate` is only a candidate label during PB12, not a promotion event
- a `Factory Candidate` is only a candidate label during PB12, not a factory creation event
- if evidence is insufficient, the item must remain local or deferred rather than being over-promoted

## Implementation Rule

Implementation may begin only after:

- PB11 package review is complete
- repository validation is complete
- the grouped backlog exists
- every backlog item has passed `Promotion Review`

After that gate, PB12 may execute only the highest-priority `Reference Quality` improvements first.

No other priority group may be implemented until the CEO checkpoint is reached.

## Reference Quality Qualification Standard

An item should be treated as `Reference Quality` only when it improves `GaryGiam.com` as a benchmark GEH reference implementation rather than merely improving the local website.

Typical signals include:

- stronger reusable repository structure
- clearer benchmark asset organization
- more reliable reference evidence capture
- more inheritable implementation discipline
- clearer separation between local content and reusable reference patterns

## Reference Value Statement

Every `Reference Quality` implementation item must explicitly record:

- `Problem`
- `Reference Value`
- `Supporting Evidence`
- `Promotion Candidate` (`Local` / `Factory` / `GEH Core`)
- `Success Metric`

This requirement exists because PB12 is no longer just improving `GaryGiam.com`.

It is improving the benchmark that future `GEH Instances` may inherit.

### Reference Test

PB12 must apply the following test throughout backloging, classification, and implementation planning:

If a new GEH Instance were installed tomorrow, would we want it to inherit this improvement by default?

Interpretation:

- if yes, it belongs in `Reference Quality` or is at least a `Factory Candidate`
- if no, it remains a `Local Improvement`
- if the evidence is promising but not yet mature enough for inheritance, it should remain a candidate rather than being over-promoted

## Factory Candidate Qualification Standard

An item should be treated as a `Factory Candidate` only when repository evidence suggests:

- repeatable output generation
- bounded inputs and outputs
- explicit operating rules
- likely reuse beyond this single instance

PB12 may identify such candidates, but may not promote them into official GEH Factories.

## GEH Core Candidate Qualification Standard

An item should be treated as a `GEH Core Candidate` only when the evidence suggests it may eventually deserve governed canonical inclusion beyond local reference value.

During PB12:

- candidate status is only an evidence marker
- no GEH Core changes are made
- no constitutional promotion is attempted

## Local Improvement Qualification Standard

An item remains a `Local Improvement` when it is useful to `GaryGiam.com` but lacks sufficient evidence, reuse value, or governance maturity for reference, factory, or core-candidate treatment.

## Deferred Qualification Standard

An item belongs in `Deferred` when:

- evidence is too weak
- cross-context validation is missing
- a separate programme is required
- authorization does not yet exist
- implementing it now would violate PB11 or the approved PB12 scope

## Knowledge Harvest

PB12 must capture reusable learnings from the implemented `Reference Quality` work.

The Knowledge Harvest should record:

- what was observed
- what changed
- why it mattered to reference quality
- what appears reusable
- what remains local
- what may later qualify as a factory or core candidate

This harvest must remain evidence-based and must not self-promote findings into canonical policy.

## CEO Checkpoint

PB12 must pause after:

- the first `Reference Quality` implementation tranche is complete
- the `Knowledge Harvest` is captured

At that checkpoint, the output should be ready for CEO review before any move into another priority group.

## Non-Goals

PB12 does not:

- modify `GEH Core`
- promote new GEH Factories
- promote GEH Core candidates into release content
- treat convenience as evidence
- skip classification and move directly into implementation

## Success Criteria

PB12 is successful when:

- the external PB11 package is fully used as the constitutional baseline
- repository findings are evidence-backed
- the backlog is grouped and prioritized clearly
- every backlog item is explicitly classified in `Promotion Review`
- only the top `Reference Quality` items are implemented first
- reusable learnings are captured in a `Knowledge Harvest`
- work pauses at the CEO checkpoint before any next priority group begins
